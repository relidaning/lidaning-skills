# README.md

Maintain the project README. It should explain **key concepts**, **core
features**, and **mechanisms** — not just commands. Someone reading it should
understand what the project is and how it works, not just how to run it.

## What a good README contains

1. **What it is** — one sentence. Purpose, not implementation.
2. **Key concepts** — the central ideas the project is built around. Domain
   terms, architectural patterns, design philosophy.
3. **Core features** — what the project can do. User-facing capabilities,
   not implementation details.
4. **Mechanisms** — how it works under the hood. Architecture overview,
   data flow, key abstractions. Diagrams welcome if they clarify.
5. **How to install and run** — exact, copy-pasteable commands. Keep this
   section short; link to detailed docs if needed.

## Rules

- **Concepts over commands** — a README should explain *why* and *what*
  before *how*. The how is the easy part.
- **Keep it current** — when a new concept is introduced or a mechanism
  changes, update the README. Don't let it rot.
- **Write for a new team member** — someone joining today should understand
  the project in 5 minutes.
- **No generated content** — don't list every file. Describe by purpose,
  not exhaustiveness.
- **Update after major changes** — new feature, new dependency, new
  architectural pattern → check if README needs a refresh.
